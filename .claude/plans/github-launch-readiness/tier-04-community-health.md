<context>
The repo has a PR template, CODEOWNERS, and dependabot [src: ls .github/], but no
CONTRIBUTING, CODE_OF_CONDUCT, SECURITY, issue forms, or CHANGELOG — all expected
of a credible public project. Depends on tier-01 because CONTRIBUTING references
the license + commercial-relicensing terms [src: plan.md D2].
</context>

<files>
- `CONTRIBUTING.md` — create; spec lifecycle, commit/scope rules, build/test
  commands, contributor license terms.
- `CODE_OF_CONDUCT.md` — create; Contributor Covenant 2.1, contact filled.
- `SECURITY.md` — create; supported versions + private reporting.
- `.github/ISSUE_TEMPLATE/bug_report.yml` — create; bug issue form.
- `.github/ISSUE_TEMPLATE/feature_request.yml` — create; feature issue form.
- `.github/ISSUE_TEMPLATE/config.yml` — create; chooser config.
- `CHANGELOG.md` — create; `cog changelog` output.
</files>

<steps>
1. `CONTRIBUTING.md`: document the three-step spec lifecycle (`/spec-plan` →
   `/spec-build` → `/spec-audit`) [src: CLAUDE.md `<workflow>`]; Conventional
   Commits with required scope [src: cog.toml `scopes`]; commands `cargo build
   --workspace`, `cargo nextest run --workspace`, `cargo clippy ... -D warnings`,
   `cargo fmt --all --check`, `cargo deny check` [src: CLAUDE.md `<commands>`].
   Add a "Contributor license terms" section: contributions are submitted under
   the project license and the contributor grants the maintainer the right to
   also license the contribution commercially (required for the dual-license
   model in plan.md D2); mark whether a formal CLA is required as an owner
   decision `<OWNER-DECISION-PLACEHOLDER>`.
2. `CODE_OF_CONDUCT.md`: fetch the Contributor Covenant 2.1 markdown from
   `https://www.contributor-covenant.org/version/2/1/code_of_conduct/code_of_conduct.md`
   and adopt it verbatim, replacing `[INSERT CONTACT METHOD]` with the owner's
   contact placeholder and keeping the attribution line [src:
   https://www.contributor-covenant.org/version/2/1/code_of_conduct/].
3. `SECURITY.md`: a "Supported Versions" table (latest minor supported) and a
   "Reporting a Vulnerability" section pointing to GitHub private vulnerability
   reporting and a contact placeholder. Keep it short.
4. Issue forms under `.github/ISSUE_TEMPLATE/` — each YAML needs top-level
   `name`, `description`, `body`; body elements use `markdown` / `input` /
   `textarea` / `dropdown` / `checkboxes` with `validations.required`
   [src: https://docs.github.com/en/communities/using-templates-to-encourage-useful-issues-and-pull-requests/syntax-for-issue-forms].
   `bug_report.yml`: version (`ariadne --version` or commit), OS dropdown
   (macOS/Windows/Linux), steps, expected, actual, logs. `feature_request.yml`:
   problem, proposed solution, alternatives. `config.yml`:
   `blank_issues_enabled: false` plus a `contact_links` entry to Discussions or
   the commercial-licensing contact.
5. Generate the changelog: `cog changelog > CHANGELOG.md` (cocogitto config is
   present) [src: cog.toml; https://docs.cocogitto.io/reference/config.html].
   If no tags exist yet it renders the full history; commit the result.
</steps>
