<verdict>
PASS. Every exit criterion is independently verified: `installers =
["shell","powershell","homebrew"]` with `tap = "mind-decay/homebrew-tap"` set;
`dist generate --check` is clean (no CI drift); `dist plan` lists all three
installer artifacts plus the required macOS arm64/x64 and Windows x64 archives;
and `docs/releasing.md` documents the tap repo, the classic `repo`-scoped PAT,
and the `HOMEBREW_TAP_TOKEN` secret.

The prior audit's blocker (F-01: the Homebrew install command and formula
filename naming `ariadne` where dist produces `ariadne-cli`) is resolved —
`docs/releasing.md:45` now commits `ariadne-cli.rb` and `docs/releasing.md:49`
installs `…/ariadne-cli`, matching the `dist plan` formula name and
`install_hint`. The only residual observation is I-01: the doc spells the tap as
`homebrew-tap` rather than dist's short `tap`. That is not a defect — Homebrew's
tap parser strips a leading `homebrew-` prefix (`repository.sub(
HOMEBREW_OFFICIAL_REPO_PREFIXES_REGEX, "")`), and the documentation states the
short form is "a shortcut for the long version `username/homebrew-foobar`", so
both forms resolve to the same GitHub repo and the documented command installs
successfully. It does not gate.

The `publish-jobs = ["homebrew"]` key is beyond the literal text of step 2 but
is the correct, necessary mechanism for the `tap` to actually publish (D4): it
is what makes `dist` emit the `publish-homebrew-formula` job in `release.yml`,
and the whole config is internally consistent under `dist generate --check`. Not
a defect.
</verdict>
