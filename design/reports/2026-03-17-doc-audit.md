# Documentation Audit Report

**Date:** 2026-03-17
**Mode:** docs-only (no Rust source code exists yet)
**Documents audited:** architecture.md, ROADMAP.md, error-handling.md, performance.md, testing.md, path-resolution.md, determinism.md, distribution.md, decisions/log.md, specs/2026-03-17-phase1a-mvp.md, specs/2026-03-17-phase1a-implementation-plan.md

## Summary

**22 findings:** 3 critical, 4 high, 9 medium, 6 low

Overall: The design documents are comprehensive and well-structured, but `architecture.md` has not been updated to reflect several later decisions (D-006, D-010), creating contradictions that would directly cause incorrect implementations. The decision log is missing ~19 concrete technical choices that appear in design docs without formal D-xxx entries.

---

## Critical

### C-01: Performance target conflict — "1-3s" vs "<10s" for 3000 files

**Documents:** `architecture.md` ("Fastest option: 3000 files in 1-3 seconds") vs `performance.md` (Time Budget table sums to ~5s; Scaling Characteristics and Benchmark Targets both say `<10s`)

**What's wrong:** architecture.md claims 1-3 seconds. performance.md's detailed model budgets ~5s and sets the target at <10s. D-001 in decisions/log.md also quotes "1-3s" as reasoning for choosing Rust — this claim predates the performance model.

**Source of truth:** `performance.md` (detailed analysis supersedes the earlier estimate)

**Fix:** Update `architecture.md` Engine Architecture section and `decisions/log.md` D-001 to reference the <10s target from performance.md.

---

### C-02: `"generated"` field in graph.json contradicts determinism.md

**Documents:** `architecture.md` Storage Format (graph.json example includes `"generated": "2026-03-17T14:30:00Z"`) vs `determinism.md` ("Remove `generated` from default output... Without `--timestamp`, graph.json has no `generated` field")

**What's wrong:** architecture.md's canonical JSON example includes a field that D-006/determinism.md explicitly removes from default output.

**Source of truth:** `determinism.md` / D-006

**Fix:** Remove `"generated"` field from the graph.json example in `architecture.md`. Add a note that `--timestamp` optionally adds it.

---

### C-03: INV-11 in testing.md uses superseded weak form

**Documents:** `testing.md` INV-11 ("edges may be in different order — compare as sets") vs `determinism.md` ("strengthened" to "byte-identical output")

**What's wrong:** testing.md still has the old version that allows set comparison. determinism.md explicitly supersedes this with byte-identical comparison.

**Source of truth:** `determinism.md`

**Fix:** Update INV-11 in `testing.md` to the byte-identical formulation.

## High

### H-01: `cargo install ariadne` vs `cargo install ariadne-graph` — D-010 not propagated

**Documents:** `distribution.md` (line 30: `cargo install ariadne`), `architecture.md` (Installation section: `cargo install ariadne`) vs D-010 in `decisions/log.md` and `ROADMAP.md` line 7 (crate name is `ariadne-graph`)

**What's wrong:** D-010 decided the crate name is `ariadne-graph`. Two documents still show the old name. distribution.md also frames the name as unresolved ("If so, alternatives: `ariadne-graph`...") when it's already decided.

**Fix:** Update all `cargo install ariadne` references to `cargo install ariadne-graph`. Rewrite distribution.md's Crate Name section to reflect D-010 as settled.

---

### H-02: architecture.md presents Phase 2 features as Phase 1

**Documents:** `architecture.md` CLI Interface (lists `ariadne update`, all `ariadne query *` commands, `ariadne views generate`) and Storage Format (`stats.json`, `views/` directory) with no phasing caveats vs `ROADMAP.md` (these are all Phase 2 deliverables)

**What's wrong:** A developer reading architecture.md would implement Phase 2 commands and outputs in Phase 1. architecture.md reads as the complete specification with no indication of phased delivery.

**Fix:** Add phase annotations to architecture.md's CLI section and Storage Format section marking which items are Phase 1a, Phase 1b, and Phase 2.

---

### H-03: Default output directory path conflict — `.ariadne/` vs `.ariadne/graph/`

**Documents:** `architecture.md` Storage Format (`.ariadne/` with files directly inside) vs Phase 1a spec (line 66: `.ariadne/graph/`) and implementation plan (line 152: `.ariadne/graph/`)

**What's wrong:** Mutually exclusive paths. If implemented as `.ariadne/graph/`, Phase 2 additions (stats.json, views/) would need restructuring. If `.ariadne/`, the spec and plan are wrong.

**Fix:** Decide canonical path and update the non-canonical document. Likely: update architecture.md to `.ariadne/graph/` for graph output files, keeping `.ariadne/` as the parent for all output types. Requires a new decision entry.

---

**What's wrong:** architecture.md is incomplete as a CLI reference.

## Medium

### M-01: Memory estimates differ by ~10x

**Documents:** `error-handling.md` ("50k files × ~500 bytes/node = ~25MB" and "50k files with 150k edges ≈ 55MB") vs `performance.md` (50,000 files → ~250-400MB)

**What's wrong:** error-handling.md uses simplified arithmetic (nodes only or nodes+edges at struct size). performance.md accounts for realistic overhead (allocator, strings, Vecs). The 50k limit justification in error-handling.md cites a 10x underestimate.

**Fix:** Update error-handling.md's memory justification to reference performance.md's figures.

---

### M-02: W005 SymlinkLoop is unreachable

**Documents:** `error-handling.md` defines W005 (SymlinkLoop — "Symlink resolves to ancestor directory") but also states "symlinks are skipped" and the walking pseudocode skips all symlinks before traversal.

**What's wrong:** If symlinks are always skipped, a symlink loop can never be detected. W005 is dead code in the error taxonomy.

**Fix:** Either define when symlink following is enabled (making W005 reachable), or remove W005 and document that symlink loops are prevented by not following symlinks.

---

### M-03: Cluster/sort pipeline stages have no error handling defined

**Documents:** `architecture.md` pipeline: "walk → read → parse → resolve → cluster → sort → output" (7 stages). `error-handling.md` "Error Handling by Pipeline Stage" covers 5 stages (Walking, Parsing, Path Resolution, Config File Parsing, Output Writing). Cluster and sort stages are missing.

**Fix:** Add error handling definitions for clustering and sorting stages in error-handling.md.

---
